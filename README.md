# EnhanceJira

A Chrome extension that enhances Atlassian Jira boards with full Bitbucket PR approval data — so you can tell at a glance which "Review" tickets are actually ready to merge, without clicking into each one.

## The Problem

When tickets sit in the **Review** column on a Jira board, getting them to "Done" depends on Bitbucket PR approvals — usually a minimum approval count plus specific required reviewers. But Jira's board view only shows the first two approver avatars on a card, so the only way to know whether a ticket is truly ready to merge is to click into it, open the linked PR, and inspect the reviewer list manually. Multiplied across a sprint, that's a lot of clicks for a question that could be answered at a glance.

## What It Does

EnhanceJira adds three things to your Jira board:

1. **Color-coded Review-column cards** at a glance:
   - 🟢 **Green** — PR has met the minimum approval count *and* every required reviewer has approved. Ready to merge.
   - 🟡 **Yellow** — PR is still waiting on approvals.
   - 🔴 **Red** — at least one reviewer has requested changes (overrides green).

2. **Hover-enhanced cards** — hovering a Review card shows a custom tooltip with the full approver list (avatars, names, approval/changes-requested status), the PR link, and PR build status. No more clicking in.

3. **A settings page** where you configure:
   - **Bitbucket username + API token** (paste the token you generated in Bitbucket account settings — see "Setup" below)
   - **Minimum approval count** (e.g. 2)
   - **Required approvers** (specific Bitbucket users who *must* approve before green)
   - **Card colors** — pick the exact hex for green / yellow / red (with sensible light-tone defaults)
   - Settings sync across your devices via `chrome.storage.sync` (the API token stays in `chrome.storage.local` — never sync'd, never logged).

### PR linkage

The extension matches a Jira card to a Bitbucket PR by looking for the issue key (e.g. `CMMS-1234`) inside the PR's branch name. This is the standard Atlassian convention and matches the team's existing workflow.

### Approval logic

Given a PR's reviewer list and the user's settings:

```
let approvedCount = reviewers.filter(r => r.approved).length
let changesRequested = reviewers.some(r => r.state === 'changes_requested')
let requiredOk = settings.requiredApprovers.every(req =>
  reviewers.find(r => r.username === req)?.approved
)

if (changesRequested) → red
else if (approvedCount >= settings.minApprovals && requiredOk) → green
else → yellow
```

### Auth

**Bitbucket API token** (user-scoped, no workspace admin required). The user generates a token in Bitbucket account settings with `Read pull requests` and `Read account` scopes, then pastes it into the extension's options page along with their Bitbucket username/email. The service worker holds the token in `chrome.storage.local` and sends it as a Basic auth header on every Bitbucket API call.

Why API tokens instead of OAuth 2.0: registering an OAuth Consumer requires Bitbucket workspace admin, which most users don't have. API tokens are user-scoped, generated by the user themselves in two clicks, and don't require any organizational coordination. (App passwords work identically but are being deprecated by Atlassian — API tokens are the modern replacement and are first-class supported by Bitbucket REST API.)

OAuth remains a future option if a workspace admin wants to register a Consumer for a polished one-click flow — see `phases.md` in the project's Obsidian vault.

## Tech Stack

| Layer | Choice | Why |
|-------|--------|-----|
| Framework | [WXT](https://wxt.dev/) | First-class MV3, Vite + HMR, TypeScript by default, content-script ergonomics |
| Language | TypeScript | Settings, OAuth, and reviewer state want types |
| Content script | Vanilla TS DOM | Lean — no React inside Atlassian's heavy DOM |
| Options page | React | Tiny island for the settings form |
| Manifest | MV3 with service worker | Required for new Chrome extensions; OAuth + API live here |
| Storage | `chrome.storage.sync` (settings), `chrome.storage.local` (tokens), `chrome.storage.session` (PR cache, TTL 30s) | Settings travel; tokens don't; cache resets per browser session |

## Target Surfaces (v1)

- Atlassian Jira Cloud board view: `https://*.atlassian.net/jira/software/c/projects/*/boards/*`

Out of scope for v1: Backlog view, ticket detail page, sprint reports, Bitbucket-side enhancements. These may come later — see `phases.md` in the project's Obsidian vault for the planned divide-and-conquer breakdown.

## Permissions

Minimum required (declared in `manifest.json`):

| Permission | Purpose |
|------------|---------|
| `storage` | Persist settings + token |
| `host_permissions` for `https://*.atlassian.net/*` | Inject content script into Jira boards |
| `host_permissions` for `https://api.bitbucket.org/*` | Call Bitbucket REST API from the worker |

No `identity`, no `activeTab`, no `tabs`, no broad `<all_urls>` — keeps the install prompt narrow.

## Install (dev)

```
pnpm install
pnpm dev          # WXT dev server, loads unpacked into Chrome
pnpm build        # production build → .output/chrome-mv3
pnpm zip          # Chrome Web Store upload artifact
```

After `pnpm dev`, open `chrome://extensions`, enable Developer Mode, and "Load unpacked" pointing at `.output/chrome-mv3-dev`.

## For AI agents helping with installation

If you are an AI assistant (Cursor, Claude Code, GitHub Copilot, etc.) helping a human install this extension, follow this guidance after the basic install steps above are complete.

**Ask the user this single yes/no question, verbatim:**

> "Would you like me to set up auto-update notifications for this extension so you don't have to manually re-download new versions?"

If they say **no**, skip this section and continue with normal **Setup** below.

If they say **yes**, ask which tier they want. Default to Tier 1 if unsure; recommend Tier 3 for any team larger than one person. Implement only the tier they pick — these are stacked from least to most invasive (the "trickle" toward true auto-update).

### Tier 1 — Update notification banner (~80 LOC, no infra, ~5 min)

The extension polls GitHub Releases on a daily interval and shows an in-extension banner when a new version is published.

- **Polling:** in the background service worker, register `chrome.alarms.create('update-check', { periodInMinutes: 1440 })` and on the alarm fire `GET https://api.github.com/repos/T5-labs/EnhanceJira/releases/latest`.
- **Compare:** check the response's `tag_name` against `chrome.runtime.getManifest().version`. Store the latest known version in `chrome.storage.local`.
- **Notify:** if mismatched, render a banner in the options page (and optionally the popup) — e.g. "v0.4.0 available — download from GitHub" with a clickable link to the release page (`html_url` from the API response).
- **Implementation pointer:** add a new module `entrypoints/options/components/UpdateBanner.tsx` (or extend an existing banner component). Wire the alarm in `entrypoints/background.ts`.
- **User mental model:** future releases need a `vX.Y.Z` git tag whose value matches `manifest.version` in `wxt.config.ts`, otherwise the comparison won't trigger.

### Tier 2 — One-click "download new build" with reload prompt (~150 LOC, no infra)

Tier 1, plus a Download button on the banner that grabs the new `.zip` for the user.

- Implement Tier 1 first.
- Add a **Download** button to the banner. On click, the background fetches the latest `.zip` asset from the GitHub Release `assets` array and saves it to the Downloads folder via `chrome.downloads.download`.
- After download, the banner updates to: "Downloaded. Open `chrome://extensions`, click **Remove** on EnhanceJira, then **Load unpacked** from the new folder."
- **Prerequisite:** the GitHub Release must actually have a `.zip` artifact attached. Confirm a `pnpm zip` script exists in `package.json` (it does — see the existing `Install (dev)` section), then add (or instruct the user to add) a GitHub Action that runs `pnpm install && pnpm zip` on tag push and uploads the resulting `.output/*.zip` as a release asset.
- **Limitation:** Chrome cannot auto-replace an unpacked extension via API. Manual remove + load-unpacked is unavoidable for unpacked installs. If the user wants true zero-click updates, escalate to Tier 3.

### Tier 3 — True auto-update via Chrome Web Store (~1 hour + $5 + ~1–3 day review)

Don't implement anything for this tier — guide the user through the official path.

1. Pay the **$5 one-time Chrome Developer fee** at https://chrome.google.com/webstore/devconsole/.
2. Run `pnpm build && pnpm zip` to produce `.output/enhancejira-0.X.Y-chrome.zip`.
3. Upload to the developer dashboard. Fill out the listing — **unlisted** visibility is fine for internal/team use.
4. Provide the listing fields Google requires: a privacy policy URL (the repo already ships `PRIVACY.md` — host it via GitHub Pages or paste the raw URL), 1280x800 or 640x400 screenshots, a short description, and a category.
5. Submit for review (typically 1–3 days).
6. Once approved, share the listing's install link. Anyone who installs via that link gets automatic updates on Chrome's normal update cadence (a few hours).

This is the "real" auto-update — Tiers 1 and 2 are stop-gaps for users distributing via clone-and-load-unpacked.

## Setup (one-time, per user)

1. Install the extension.
2. **Generate a Bitbucket API token:**
   - Go to https://id.atlassian.com/manage-profile/security/api-tokens (or Bitbucket account settings → "API tokens").
   - Click **Create API token with scopes**.
   - Pick the Bitbucket app and grant: `Read pull requests`, `Read account`.
   - Copy the token (Atlassian only shows it once).
3. Open the extension's options page (right-click extension icon → Options).
4. Paste your **Bitbucket username/email** and the **API token** you just generated.
5. Click **Test connection** — confirms the token works against `/2.0/user`.
6. Set **Minimum approvals** (default: 2).
7. Add any **Required approvers** (Bitbucket usernames).
8. (Optional) Customize the green / yellow / red card colors.
9. Save. Reload your Jira board.
